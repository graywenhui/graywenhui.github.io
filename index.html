概述

xv6文件系统分为7层，从底层网上依次是磁盘（Disk）、缓冲区高速缓存（Buffer cache）、日志（Logging）、索引结点（Inode）、目录（Directory）、路径名（Pathname）、文件描述符（File descriptor）。

磁盘读取和写入virtio硬盘上的块（磁盘被分为一个个块）。缓冲区高速缓存缓存经常访问的块并同步对它们的访问。日志可以将多个更新操作包装到一个事务中，确保遇到崩溃时自动更新这些块，即保证这些更新操作要么全部成功，要么全部失败，它可以使用日志信息进行数据的还原和恢复。索引结点提供单独的文件，每个文件表示为一个索引结点，包含唯一的索引号（i-number）和保存文件数据的块。目录层将每个目录实现为一个特殊的索引结点，内容时一系列的目录项，每个目录项包含一个文件名和索引号，这点可以参照之前的lab用到的一些调用，比如fstat和相关的函数，就可以理解其中的意思了。路径名提供分层路径名，通过递归查找来解析它们。文件描述符是最高层的抽象，抽象了许多资源。

为什么会有sleep和wakeup

原先的信号量同步机制，v操作和p操作可能是这样的

在这样的情况下，（消费者）大部分时间都是在while循环中繁忙等待，我们希望有一种行为，它能够在等待的时候释放cpu，直到有其他调用将它唤醒，这样就不会将cpu浪费的循环中了。假设这样的一对调用（等待，和唤醒）为sleep和wakeup，并且它们的参数都只有一个semaphore类型的变量。

这时候pv操作可能变成这样：

这样又带来了一个新的问题，可能会丢失唤醒。如果有个进程调用p，p发现s->count为0，接着要执行sleep(s)，但在执行sleep之前，在这中间的时间段中，有另一个进程调用了v，将count++并且掉用wakeup，这时第一个进程还没开始调用sleep，所以它调用以后会需要等待下一个wakeup，这就丢失了之前的wakeup。

根本原因是，v在不该运行的时候运行，违反了不变量-p只有在s->count==0的时候才sleep，p在判断并且进入sleep之间，其他进程是不应该调用到wakeup的，所以我们立马想到应该加一个锁或者改变锁的位置，于是代码可能变成了这样。

这又带来了一个新的问题，大家可以很快发现这一定会死锁。因为p在sleep的时候是获取着锁的，这意味着没有一个进程能成功在V中走到wakeup这步。

所以我们需要更改sleep内部的内容。我们需要传递一个在sleep内需要先释放的锁，以便其他进程能够成功调用V，并且在需要睡眠的进程没有sleep之前将一直阻塞，知道需要睡眠等待的进程进入sleep并释放锁以后，才能获取到锁。在调用完wakeup后，睡眠的进程在被唤醒前又会再次获得锁。需要注意的是，这里传的锁就是之前已经传的结构体中的一个成员，为什么要传呢。因为其他的情况中，sleep的第一个参数可能没有锁，也可能需要获得的是另一个锁，所以还是需要传入一个锁的参数的。

上面的内容也就解决了为什么sleep内部要先释放锁，被唤醒以后再获得锁的问题。

Thread switching

假如有两个线程相互间切换，宏观上需要走这么一个过程：先从第一个用户进程进入内核，保存该进程的状态并运行该线程的内核线程，然后从该内核线程切换到第二个用户进程对应的内核进程，该内核进程暂停自己，恢复对应用户进程的状态（第二个用户进程），回到用户进程中。

更细致一点：定时器中断导致CPU从用户态进入内核态，trampoline将用户寄存器保存到进程对应的trapframe里；然后内核会执行usertrap代码，执行相应的中断处理程序，完了后会调用swtch函数；该函数保存用户进程对应的内核线程的寄存器到context对象。

这里有一个需要注意的地方就是，我们现在是在调用swtch函数的返回状态，不过不是在上面的swtch函数返回，而是在之前CPU调用的返回状态，之前调用的结果是CPU运行了我们刚刚中断了的进程。这么说可能有点绕，可以理解为因为汇编代码修改了ra的内容，所以函数返回在时间上有延迟，现在返回的是之前的运行的swtch函数，而我们刚刚运行的swtch函数其实还没有返回。

继续看代码的内容，这时候从swtch函数调用返回，下一步执行c->proc = 0修改状态，表明这时候CPU没有在运行进程。然后释放锁，这里释放的锁是之前在yield里获取的锁，在yield里我们没有完全完成yield进入runable时有别的CPU能运行它，现在已经完全进入runable状态了所以我们释放锁。

p->lock的两个作用：

  1. 让出CPU有几步，改状态、保存上下文、停止使用当前进程的栈等，第一个作用就是确保这些步骤的原子性
  2. 启动进程的时候也需要保证原子性，让别的CPU看不到它，所以需要加锁，这里还需要注意的是，切换进程的时候，还要关闭中断，因为如果在切换到一半的时候产生了中断，会导致不好的后果，比如可能寄存器没切换完，造成混乱等。

现在我们在循环中，先获取锁，然后开始切换进程，修改完状态后，再次调用swtch函数，这里切换到这次循环中命中的进程。这里注意，之前在sched里调用swtch是swtch(&p->context, &mycpu()->context)，而在这里是swtch(&c->context, &p->context)，也就是这里是保存CPU调度进程的上下文，恢复命中进程的上下文。这里恢复的ra就会是这个进程之前被调用swtch(&p->context, &mycpu()->context)返回的地址。

这可能会带来一个问题，我们能看到其实每次调用swtch返回到的其实都是另一个之前的swtch的调用，那第一次swtch调用的时候会返回到哪里呢。看它返回的地址只要关注ra寄存器，第一个进程启动后会调用allocproc，其中会设置好ra寄存器和sp寄存器。

即第一个swtch返回后会执行forkret中的内容。

STL

STL全称为Standard Template Library，即标准模板库。

带给我们泛型编程的思想，即实现可重用、通用的代码。

STL提供了六大组件：

  1. 容器：各种数据结构，可以看成是class template
  2. 算法：各种常用算法，sort，copy等，可以看成是function template
  3. 迭代器：泛型指针，重载指针相关操作
  4. 仿函数：重载了operator()的class
  5. 适配器：修饰容器或仿函数或迭代器接口的东西，类似于容器，但底层是由其他容器实现的。
  6. 配置器：负责空间配置与管理，是一个实现了管理内存空间的class template

STL的序列式容器有以下几个

  1. array数组（内置）
  2. vector (heap) (priority_queue)
  3. list
  4. slist
  5. deque (stack) (queue)

首先最外层循环512次是因为每个页表都有512个PTEs，对每个PTE先看它的有效性，因为有三级页表，所以根据页表等级来打印输出。
这里PTE2PA(PTE)是xv6定义的一个宏#define PTE2PA(pte) (((pte) >> 10) << 12),这个宏的作用是将PTE转化成PA，先右移10位将标志位全移除，然后左移12位是因为有12位的偏移。
接着打印出信息，然后判断是否到了页表的结尾，如果没到就递归调用下一级页表。

此外要在第一个进程运行的时候做这件事，也就是内核刚开始运行的时候，所以在exec()中进程被创建好之后加上if(p->pid==1) vmprint(p->pagetable);这样在第一个进程开始时就会开始调用。
